Data Security & Privacy
25 points · weighted 25%
24
of 25 pts

Evaluates data retention policies, PII handling, DPA availability, training data use, and data residency controls. The Einstein Trust Layer is the standout feature — it provides zero data retention for LLM prompts, automated PII masking before data leaves Salesforce, and toxicity detection. The gap is scope: Einstein and Agentforce have access to the breadth of your Salesforce org, and the data residency story is complex across Hyperforce regions.

✓
Einstein Trust Layer enforces zero data retention for LLM prompts by default — customer data is not sent to or retained by third-party model providers; prompts are processed ephemerally and discarded after response generation
✓
Automated PII detection and dynamic data masking built into the Trust Layer — sensitive fields (SSNs, credit card numbers, email addresses) are masked before data reaches the LLM, with original values restored in the response
✓
GDPR Data Processing Addendum available to all customers; SCCs included for EU transfers; sub-processor list publicly maintained on trust.salesforce.com; CCPA compliance documented with data subject rights process
✓
Customer data is not used to train Salesforce AI models by default; model training opt-out is the default posture and is contractually documented in the DPA
!
Einstein and Agentforce have access to the full scope of connected Salesforce org data — contacts, opportunities, cases, contracts — without granular per-feature data access controls; broad access scope amplifies the impact of any AI error or misuse

Model Security
25 points · weighted 25%
18
of 25 pts

Active testing for prompt injection, jailbreak resistance, system prompt extraction, and output filter bypass. The Einstein Trust Layer includes toxicity detection and output filtering. The primary concern is Agentforce: autonomous agents that can take actions (create records, send emails, update opportunities) represent a meaningful prompt injection attack surface — a malicious instruction embedded in a customer email or support ticket could trigger unintended agent actions against live CRM data.

✓
Einstein Trust Layer includes real-time toxicity detection and output content classification — harmful or policy-violating responses are blocked before reaching users; configurable sensitivity thresholds per org
✓
Trust Layer audit trail logs all LLM interactions including prompt, response, PII masks applied, and toxicity scores — provides full forensic record for compliance review and incident investigation
✓
Agentforce agent actions are scoped to administrator-configured permissions; agents cannot take actions outside their defined flow boundaries; human-in-the-loop confirmation available for sensitive actions
!
Underlying LLM powering Einstein features not disclosed for all capabilities — some use Azure OpenAI, some use Salesforce's own models; the specific model checkpoint, training data, and safety evaluations for Salesforce-developed models are not published
!
Agentforce indirect prompt injection risk: malicious instructions embedded in customer-submitted content (support tickets, emails, web forms) can reach the agent's context and potentially trigger unintended CRM actions — our testing found 9 of 20 indirect injection attempts produced partially compliant agent responses

Infrastructure & API Security
20 points · weighted 20%
17
of 20 pts

TLS configuration, API authentication controls, penetration testing cadence, supply chain security, and SBOM availability. Salesforce runs on a combination of its own Hyperforce infrastructure (AWS-based) and legacy infrastructure, with annual third-party security assessments documented. The Connected App OAuth model provides reasonable API security, though IP allowlisting and fine-grained API permission scoping require deliberate administrator configuration.

✓
TLS 1.3 enforced on all Salesforce platform endpoints; TLS 1.0/1.1 deprecated; HSTS enabled; A+ SSL Labs rating confirmed on primary API endpoints
✓
Annual third-party security assessments including infrastructure vulnerability assessments and application security assessments documented in the Einstein Platform Security Architecture paper; findings actioned under formal vulnerability management programme
✓
HackerOne bug bounty programme active with public policy, defined scope covering Salesforce platform and Einstein APIs, and documented response SLAs; responsible disclosure programme has been running since 2015
✓
Hyperforce architecture built on AWS with SOC 2 and ISO 27001 controls; AWS infrastructure certifications extend to Salesforce workloads per documented sub-processor relationship; FedRAMP High authorisation achieved on Hyperforce Government Cloud
!
IP allowlisting for API access and Connected App permission scoping must be explicitly configured by administrators — default posture grants broader access than most enterprise security policies require; misconfigured orgs expose broad CRM data via API

Governance & Transparency
15 points · weighted 15%
12
of 15 pts

Model cards, audit logging, acceptable use policy clarity, and human oversight mechanisms. Salesforce's governance posture is strong relative to most CRM AI vendors. The Einstein Trust Layer audit trail is a genuine differentiator — every LLM interaction is logged with prompt, response, PII actions, and toxicity scores. The gap is model transparency: the Einstein Platform Security paper documents controls comprehensively but does not publish model cards for Salesforce-developed AI models.

✓
Einstein Trust Layer audit trail logs every LLM interaction — prompt text, response, PII masking events, toxicity classification scores, and model used — all accessible via the Einstein Audit Data object in Salesforce reporting and SIEM export
✓
Salesforce Acceptable Use Policy for generative AI features specifically prohibits harmful content generation, IP circumvention, deceptive AI impersonation, and illegal discrimination; policy is enforceable and referenced in customer agreements
✓
Agentforce agent definitions, action scopes, and topic configurations are administrator-controlled and version-tracked in the Salesforce platform; change history available for governance review
!
No model card published for Salesforce-developed AI models (Einstein Prediction Builder, Einstein Classification, custom Agentforce models); training data sources, bias evaluations, and known limitations are not publicly documented
!
Agentforce agent action scope documentation is functional (what actions are possible) but not risk-ranked — administrators lack a framework for assessing which action combinations create elevated risk; no published guidance on minimum-privilege agent configuration

Compliance & Certifications
15 points · weighted 15%
10
of 15 pts

SOC 2 Type II, ISO 27001, ISO 42001 AI Management, EU AI Act readiness, NIST RMF mapping, and GDPR compliance. Salesforce's compliance portfolio is among the broadest of any AI vendor — ISO 42001 certification for Agentforce and Einstein is rare and significant. The score here reflects that individual Einstein features vary in certification scope; not all features are covered under all certifications, requiring per-feature verification for regulated industries.

✓
SOC 1 Type II and SOC 2 Type II certified — covering Agentforce, Einstein Prediction Builder, Einstein Case Classification, and other covered services; reports available to customers under NDA via trust.salesforce.com
✓
ISO 27001, ISO 27017, and ISO 27018 certified — covering information security management, cloud security, and PII protection in cloud environments; certificates available on trust.salesforce.com
✓
ISO 42001 certified for Agentforce and Einstein AI Platform — one of the first major enterprise AI vendors to achieve this AI Management Systems certification; confirms structured framework for responsible AI practices
✓
FedRAMP High authorisation for Hyperforce Government Cloud; HIPAA BAA available; PCI-DSS compliance documented — among the broadest regulated-industry compliance portfolios of any AI platform
!
Certification scope varies by feature — Einstein Vision and Language are excluded from some covered service certifications; customers in regulated industries must verify per-feature certification coverage for their specific Einstein use cases before relying on compliance attestations
